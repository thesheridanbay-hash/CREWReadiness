## What it is

AI-powered training for landscaping companies. Owners turn SOPs / voice / field-mistake
photos into short, Duolingo-style bilingual lessons; crews complete them with an AI
teaching loop. Multi-tenant SaaS, live on Vercel.

Stack: Next.js 16 (App Router) · React 19 · TypeScript · Drizzle + Neon Postgres (FORCE
RLS) · Better Auth (org plugin) · Inngest · Vercel Blob · Stripe · Vitest + Playwright.

## Core conventions (these are uniform — follow them)

- **Server actions** (`actions/*.ts`): every action is `"use server"` → zod parse →
  `guard<Result<T>>()` → `getSession()` + role check → `scoped(auth, tx => ...)`. All 14
  follow this exactly. Copy an existing one (e.g. `actions/coaching.ts`) when adding one.
- **Tenant isolation**: never query outside `lib/db/scoped.ts`. `scoped(session, tx)` for
  requests, `scopedForJob(jobRow)` for Inngest. Every tenant table has `companyId` + FORCE
  RLS. The runtime DB role is non-owner and cannot bypass RLS.
- **Errors**: typed `Result<T>` envelope from `lib/errors` (D15). Zod at every boundary.
- **Imports**: `@/` alias (maps to repo root today). No deep `../../..` relative imports —
  EXCEPT `lib/auth/auth.ts`, which uses relative imports to `db/` ON PURPOSE (the Better
  Auth CLI loads it outside the bundler). Do not "fix" those to `@/`.

## Two schemas (important)

- `db/schema.ts` — application tables. `drizzle.config.ts` points here. Managed by
  `drizzle-kit push`.
- `db/auth-schema.ts` — Better Auth tables (`user`, `session`, `organization`, `member`).
  Owned by the Better Auth CLI, NOT in `drizzle.config.ts`. Regenerated by the Better Auth
  generate command, not drizzle.
- `db/tenant-tables.ts` MUST stay in sync with the tenant list in `db/rls.sql`.

## DB setup runbook (the one-command version lands as scripts/db-setup.sh)

```
1. drizzle-kit push                 # app tables (as OWNER role / DATABASE_URL_OWNER)
2. Better Auth generate             # auth tables (owns db/auth-schema.ts)
3. psql -f db/rls.sql               # FORCE RLS + policies (as OWNER)
4. psql -f db/indexes.sql           # purpose-built indexes
5. psql -f db/sql/<dated>.sql       # incremental patches, in date order
```

## Target layout (feature-based — in progress)

```
src/
  app/          # Next router (thin handlers) — routes stay under app/ (Next requirement)
  app-shell/    # sidebar, mobile-*, notification-bell, language-switcher (compose features)
  features/<domain>/  { actions, queries, ui, + domain logic }   # one folder per domain above
  shared/       { ui (shadcn), components (leaf), errors, db (scoped+client), config, store }
  db/           # schema.ts + auth-schema.ts + rls.sql + indexes.sql + tenant-tables.ts + sql/
  inngest/      # client + 7-function registry (route stays at src/app/api/inngest)
```

Boundary rule (enforced by eslint-plugin-boundaries once Phase F lands): features import
each other only via a defined surface; `shared/` never imports `features/`; only
`app/`, `app-shell/`, and root `proxy.ts` may import features.

## Where to add a new feature

1. `src/features/<name>/` with `actions.ts` (copy the `guard/Result/scoped` pattern),
   `queries.ts`, `ui/`.
2. Route(s) under `src/app/...` that import from the feature (keep handlers thin).
3. New tenant tables → `db/schema.ts` + add to `db/tenant-tables.ts` AND `db/rls.sql`.
4. `npm run typecheck && npm run lint && npm run test:unit` before you commit.
